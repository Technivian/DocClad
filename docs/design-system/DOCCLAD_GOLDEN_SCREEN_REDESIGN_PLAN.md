# DocClad Golden Screen Redesign Plan — Cockpits + Code Impact

Status: **PLAN ONLY — not approved, not implemented.** Companion to [`DOCCLAD_LEGAL_WORK_ENGINE_PLAN.md`](./DOCCLAD_LEGAL_WORK_ENGINE_PLAN.md) (§4 and §6 of the brief).

## 4. Cockpit redesign plan — NDA, MSA, DPA

All three cockpits already ship the visual pattern this section would otherwise be proposing from scratch (see §0 of the main plan). What follows documents the **current, confirmed** structure of each — so it's a real baseline, not a wishlist — and calls out the small number of genuine remaining gaps per cockpit. Every `id`/`data-*` attribute listed under "must remain untouched" is load-bearing: it's read by that cockpit's own JS, by the server-side clause-id scheme (`_dpa_document_sections()`-equivalent functions), and by the post-generation Contract Workspace pages.

### NDA cockpit (`theme/templates/contracts/nda_workflow_builder.html`, route `/contracts/new/nda/`)

- **Left input column**: numbered panel sections (Counterparty and ownership; Purpose and confidentiality scope; further clause-driving fields) with required/optional pills, feeding `FieldDefinition`s for the seeded NDA `WorkflowTemplate`.
- **Center draft/document preview**: bounded document canvas, sticky `.nda-draft-topbar` header with a live `#nda-draft-state` chip ("Self-serve ready" / "Legal review required" — client-computed from the same conditions as `nda_workflow.py`'s `has_risk`, not yet a persisted field, see main plan §2.1), numbered clause badges, per-clause source-label pills.
- **Right governance rail**: grouped into Template & Playbook / Risk & Compliance / Approval & Audit, icon medallions, reads `NDA Self-Serve Playbook` / clause library count / approval route from the seeded `ApprovalRoute` rows.
- **Readiness states**: required-field completion tracked client-side against the same field set the server validates on POST; readiness pct not yet shown as a header badge the way DPA has one (DPA-only feature today — worth deciding whether to add it to NDA/MSA for consistency, low-risk visual addition if so).
- **Route preview**: static seeded approval route, e.g. Contract Owner → Signature (self-serve) or → Legal → Signature.
- **Risk/playbook signals**: `#nda-risk-signal-list` renders `detect_nda_risk_signals()` output live as fields change (confidentiality period, personal data, residual knowledge, governing law).
- **Must remain untouched**: `nda-workflow-form`, `nda-draft-doc`, `nda-template-body-data`, `nda-risk-signal-list`, `nda-gov-approval-route`, `nda-gov-approval-reasons`, `nda-gov-risk-checks`, `submit-nda-btn`, and every clause id (`purpose`, `confidential-information`, `confidentiality-obligations`, `term`, `permitted-recipients`, `personal-data`, `governing-law`, `residual-knowledge`, `injunctive-relief`, `generated-nda-draft`), plus the `_validate_nda_submission` coercion logic and `create_nda_workflow_instance` model-creation set.

### MSA cockpit (`theme/templates/contracts/msa_workflow_builder.html`, route `/contracts/new/msa/`)

- **Left input column**: `.msa-guide-panel` mini-nav over 5 sections (Contract setup, Commercial terms, Service scope, Legal positions, Review triggers) plus `.msa-command-panel` — MSA's distinct 5-step stage tracker (Intake → Draft → Commercial review → Legal approval → Signature), a structural feature the other two cockpits intentionally don't have.
- **Center draft/document preview**: same document-canvas/numbered-clause/source-pill pattern as NDA/DPA; MSA's 3 static toolbar pills ("Template source locked", "Playbook matched", "Fallbacks ready") live in its sticky `.msa-draft-topbar`.
- **Right governance rail**: same 3-group pattern, icon medallions.
- **Readiness states**: `msa-primary-status` / `msa-guide-count` track completion against the confirmed 16 required fields; no header progress-pct badge (same gap as NDA, see above).
- **Route preview**: `#msa-command-route` reflects the seeded 5-step MSA approval route, with reasoning text reacting to the Finance-threshold and non-standard-liability-cap signals (the route *steps* themselves are still static — see main plan §2.2).
- **Risk/playbook signals**: `#msa-risk-signal-list` renders `detect_msa_risk_signals()` live — value threshold, liability cap, personal data, auto-renewal, IP ownership, governing law.
- **Must remain untouched**: `msa-workflow-form`, `msa-draft-doc`, `msa-template-body-data`, `msa-command-stage`, `msa-command-risk`, `msa-command-route`, `msa-command-action`, `msa-next-action`, `msa-primary-status`, `msa-guide-count`, `msa-risk-signal-list`, `msa-gov-approval-route`, `msa-gov-approval-reasons`, `msa-gov-risk-checks`, `submit-msa-btn`, clause ids (`services`, `fees-payment`, `term-renewal`, `liability`, `intellectual-property`, `data-protection`, `governing-law`, `generated-msa-draft`), plus `_validate_msa_submission` and `create_msa_workflow_instance`.

### DPA cockpit (`theme/templates/contracts/dpa_workflow_builder.html`, route `/contracts/new/dpa/`)

- **Left input column**: numbered panels (Basic Details, Privacy Details, Legal Position) driving the 9 confirmed required fields.
- **Center draft/document preview**: the most complete of the three today — has a header progress pill (`dpa-header-progress-fill`, `dpa-progress-pct`, `dpa-progress-copy`) the other two lack, plus an AI action bar (Suggest missing values / Suggest fallback clause / Explain clause / Generate DPA summary / Compare to playbook) that is DPA-specific and intentionally not duplicated onto MSA/NDA (per the reference pattern doc, each cockpit keeps its own structural character).
- **Right governance rail**: same 3-group pattern; DPA additionally has a bottom Draft Readiness strip with a visual route track (`dpa-readiness-fields`, `dpa-readiness-missing`, `dpa-readiness-risk`, `dpa-readiness-approvals`, `dpa-readiness-blockers`) — the most complete readiness surface of the three, and arguably the pattern NDA/MSA should eventually match rather than DPA being simplified.
- **Readiness states**: fully wired — the one confirmed gap is that **DPA's risk signals never write `contract.risk_level`** (main plan §2.1 and mapping doc), so the readiness strip's risk-summary line is accurate about *signals* but there's no persisted equivalent of MSA/NDA's self-serve flag for this type.
- **Route preview**: seeded Contract Owner → Legal → DPO → Signature route; `dpa-gov-approval-route` / `dpa-gov-approval-reasons` render live reasoning.
- **Risk/playbook signals**: `#dpa-risk-signal-list` renders `detect_dpa_risk_signals()` live — cross-border transfer, DPO contact, subprocessors, special categories, breach-notification window.
- **Must remain untouched**: `dpa-workflow-form`, `dpa-draft-doc`, `dpa-template-body-data`, `dpa-progress-pct`, `dpa-progress-copy`, `dpa-header-progress-fill`, `dpa-gov-risk-checks`, `dpa-risk-signal-list`, `dpa-gov-approval-route`, `dpa-gov-approval-reasons`, `dpa-readiness-fields`, `dpa-readiness-missing`, `dpa-readiness-missing-list`, `dpa-readiness-risk`, `dpa-readiness-approvals`, `dpa-readiness-blockers`, `submit-dpa-btn`, clause ids (`processing-details`, `international-transfers`, `subprocessors`, `security`, `breach-notification`, `governing-law`, `generated-dpa-draft`), plus `_validate_dpa_submission` and `create_dpa_workflow_instance`.

### Cross-cockpit recommendation (not committed, needs your call)

The only *structural* gap across all three is the missing DPA `risk_level` write and the inconsistent readiness-progress-badge coverage (DPA has one, MSA/NDA don't). Both are small, template/service-local fixes, not a redesign. Everything else asked for in the brief's §4 is already built and consistent across the three cockpits.

### What must not change anywhere in this plan

Every route, URL name, and view class listed in the mapping and cockpit sections above. Every CSRF token pattern. `can_access_contract_action` permission semantics (`contracts/permissions.py`) — VIEW/COMMENT/AI require active org membership, EDIT requires Owner/Admin or the contract's creator; nothing in this plan proposes loosening or tightening that. The `workspace_mode` containment policy — no new UI in this plan needs to branch on it, and if a future phase decides one does, that phase owns writing its own preservation test per `tests/test_workspace_mode_containment.py`'s convention. The append-only, hash-chained `AuditLog` write path. Every `id`/`data-*`/clause-id enumerated per cockpit above.
