# CLM One Contract Type Mapping

Status: **PLAN ONLY — not approved, not implemented.** Companion to [`CLMONE_LEGAL_WORK_ENGINE_PLAN.md`](./CLMONE_LEGAL_WORK_ENGINE_PLAN.md) (§3 of the brief).

## How to read this table

Every cell is either a confirmed fact from the current codebase (cited) or explicitly marked **not confirmed** / **gap**. Nothing here is invented to make the table look complete — a wrong "required fields" or "risk checks" claim in a legal-tech product's planning doc is worse than an honest blank.

Two tiers exist today, not one uniform pattern:

- **Tier 1 — dedicated cockpit** (NDA, MSA, DPA): split-screen builder, own service module, rule-based `RiskSignal`s, seeded `ApprovalRoute`.
- **Tier 2 — generic governed form** (SOW, Supplier/`VENDOR`, Amendment/`AMENDMENT`, SaaS/`SAAS`, Custom/`OTHER`): one shared `ContractCreateView`, `governance_panel` (template, clause count, approval-route preview, risk summary) and `launch_setup_map` (server-side required-field policy from `contracts/services/contract_policies.py`), no split-screen document canvas.

Note on naming: there is no `SUPPLIER_AGREEMENT` enum value in `Contract.ContractType`. "Supplier Agreement" is the UI label (`CARD_TITLES`, `contract_launch_setup.py`) for the `VENDOR` enum value. Likewise "Amendment" in the brief maps to the `AMENDMENT` enum, UI-labeled "Addendum" on the entry-card grid.

## What this table implies for phasing

Promoting SaaS Agreement or Supplier Agreement to Tier 1 (a dedicated cockpit, like DPA/MSA/NDA) is real net-new scope — a new service module, new `FieldDefinition`/`WorkflowTemplate` seed data, a new risk-signal catalogue, a new cockpit template. That's explicitly **not** in this round; see Phase F in the main plan. This round's job for Tier 2 types is closing the "needs verification" gaps above (generated-output confirmation) and, if you approve it, adding SaaS Agreement to the entry-card grid so it's discoverable the same way Supplier Agreement and Amendment already are — a small, additive, visual-only change (`ENTRY_CARD_TYPES` + `CARD_META`/`CARD_TITLES`/`CARD_DESCRIPTIONS` entries in `contract_launch_setup.py`, plus one icon in the template) with no risk-logic implications, since Tier 2 types don't have type-specific risk logic to build yet either way.
